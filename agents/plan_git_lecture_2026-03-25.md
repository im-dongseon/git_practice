## 1. 목적

이번 계획의 목표는 현재 Git 강의 자료의 강점을 유지하면서, 리뷰에서 확인된 혼선 요소와 설명 공백을 체계적으로 보완하는 것이다.

강의는 이미 실습 흐름과 협업 맥락이 잘 살아 있으므로 전면 재구성보다는 다음 네 가지에 집중한다.

- 명령어/용어의 일관성 정리
- Git 내부 동작 모델 보강
- 원격 협업 안전 수칙 보완
- 부록 및 표현 품질 개선

즉, 새 강의를 다시 만드는 것이 아니라 **현재 강의를 더 정확하고, 더 안전하고, 더 배우기 쉽게 다듬는 후속 정비 작업**으로 본다.

---

## 2. 수정 원칙

### 유지할 점

- 실습 중심 구성
- 기능 개발 → 병합 → 충돌 → 이력 정리 → 복구로 이어지는 실무형 흐름
- 짧은 설명 뒤 명령어와 실습을 배치하는 문서 스타일

### 보완할 점

- 동일 주제는 같은 용어와 최신 명령어 기준으로 통일
- 위험한 명령은 “언제 쓰면 안 되는지”까지 함께 설명
- 로컬 작업과 원격 협업 상황을 명확히 구분
- 부록 문서는 본편과 연결되도록 맥락을 보강

## 4. 파일별 수정 계획

### 4-1. `02_git.md`

**목표**  
입문자가 이후 문서들을 이해할 수 있도록 Git의 기본 모델을 먼저 잡아준다.

**수정 항목**

1. `working directory / staging area / commit history` 관계를 설명하는 소단락 추가
2. 간단한 텍스트 도식 또는 표 추가
3. 필요 시 `working tree / staging area / local repository` 구조를 보여주는 보조 이미지를 `image/` 폴더에 저장해 활용
4. 이후 `restore`, `reset`, `revert`, `stash`가 각각 어느 층위에 영향을 주는지 연결 문장 추가

**기대 효과**

- 뒤 문서에서 명령어를 외우는 대신 동작 원리로 이해하게 됨
- reset/revert/stash 구분이 쉬워짐

---

### 4-2. `03_start_new_feature.md`

**목표**  
기본 개발 흐름 문서의 표현 정확도를 높이고, merge/cherry-pick 설명을 더 현실적으로 다듬는다.

**수정 항목**

1. `merge` 설명에서 “영역을 철저히 분리해야 한다”는 인상을 줄 수 있는 표현 완화
2. 충돌이 발생할 수 있으나, 해결 가능하다는 방향으로 설명 수정
3. 필요하면 merge와 cherry-pick의 사용 예시를 한 줄씩 더 명확히 정리

**기대 효과**

- 초보자가 merge를 불필요하게 위험한 작업으로 오해하지 않음
- 후속 conflict 문서와 연결이 자연스러워짐

---

### 4-3. `04_conflict.md`

**목표**  
충돌 해결 문서를 “충돌 인지”에서 “결과 검증”까지 닫힌 흐름으로 완성한다.

**수정 항목**

1. 충돌 해결 후 확인 단계 추가
   - `git status`
   - `git log --graph --oneline --all`
2. “둘 중 하나 선택” 외에 “두 변경을 재조합하는 방식”도 해결 방법으로 설명
3. VSCode UI 해결 방식과 수동 해결 방식의 차이를 짧게 정리

**기대 효과**

- 수강생이 충돌 해결 완료 여부를 스스로 검증할 수 있음
- conflict를 단순 선택 문제가 아닌 통합 작업으로 이해하게 됨

---

### 4-4. `05_rebase_squash.md`

**목표**  
이력 재작성의 장점뿐 아니라 위험성과 사용 조건까지 분명히 전달한다.

**수정 항목**

1. `rebase`가 커밋 해시를 바꾸는 이력 재작성임을 명시
2. 이미 공유된 브랜치에서 무분별한 rebase가 왜 위험한지 설명
3. 필요 시 “개인 브랜치/PR 정리 단계에서 주로 사용”이라는 운영 가이드 문장 추가

**기대 효과**

- 실무에서 rebase를 안전하게 쓰는 기준이 생김
- 이력 정리와 협업 안전성 사이 균형을 잡을 수 있음

---

### 4-5. `06_reset_revert.md`

**목표**  
되돌리기 관련 문서를 최신 Git 사용 흐름 기준에 맞게 재정리하고, 구방식 명령과의 관계까지 함께 설명한다.

**수정 항목**

1. 단일 파일 되돌리기 설명을 `git restore <file>` 기준으로 재작성
2. 현재 사용 중인 `git checkout <file>`은 구방식(legacy)으로 소개하고, 파일 대상/브랜치 대상에 따라 동작 의미가 달라졌음을 설명
3. `git checkout`이 `git switch`와 `git restore`로 분리된 이유를 짧게 설명
4. 작업 디렉토리 복구에서 `git reset`을 사용하던 과거 방식은 참고용으로만 다루고, 현재 권장 흐름은 `git restore` 중심으로 정리
5. 신규 방식과 과거 방식을 비교하는 표 추가
6. `git reset --hard` 경고 문구 강화
7. “실습 저장소에서만 충분히 이해하고 사용할 것” 같은 안전 문장 추가
8. `reset`과 `revert` 판단 기준을 더 눈에 띄게 정리
**기대 효과**

- 초보자 혼선을 줄이고, 위험 명령어 오사용 가능성을 낮춤

---

### 4-6. 강의 전반 명령어 현대화 정리

**목표**  
강의 자료 전체에서 `git checkout` 사용 사례를 점검하고, 목적에 맞는 최신 명령어(`git switch`, `git restore`)로 정리한다.

**수정 항목**

1. 강의 문서 전체를 검색해 `git checkout` 사용 위치를 목록화
2. 브랜치 변경/생성 목적의 예시는 `git switch`, `git switch -c` 기준으로 변경
3. 파일 복구 목적의 예시는 `git restore` 기준으로 변경
4. 구방식 예시는 “예전 방식” 또는 “참고” 형식으로만 보조 표기
5. 문서 전체에서 `git reset`이 작업 디렉토리 복구 용도로 사용된 부분이 있다면 최신 흐름과 충돌하지 않게 조정

**기대 효과**

- 문서 간 명령어 기준이 통일됨
- 학습자가 `git checkout`의 이중 역할 때문에 혼란을 겪지 않게 됨

---

### 4-7. `98_tip.md`

**목표**  
실무 팁 문서의 안전성과 설명 밀도를 높인다.

**수정 항목**

1. `git stash pop` vs `git stash apply` 차이 설명 보강
2. `git commit --amend` 후 원격 반영 시 `git push --force-with-lease` 권장 문장 추가
3. amend는 “이미 공유된 커밋에는 주의”라는 경고 추가

**기대 효과**

- 팁 문서가 단순 편의 기능 모음이 아니라 실무 안전 가이드 역할도 하게 됨

---

### 4-8. `99_etc.md`

**목표**  
부록 문서를 본편 학습 흐름과 더 자연스럽게 연결한다.

**수정 항목**

1. `Git Flow`와 현재 강의의 흐름(feature branch/GitHub Flow에 가까운 방식) 차이 설명 추가
2. 커밋 메시지 규칙과 본편 실습 커밋 예시를 연결하는 문장 보강
3. 오탈자 및 문장 다듬기 반영

**기대 효과**

- 학습자가 참고 링크를 “외부 부가 자료”가 아니라 현재 강의의 연장선으로 이해하게 됨

---

### 4-9. 전반 공통 정리

**목표**  
문서 전체에서 표현과 표기 일관성을 맞춘다.

**수정 항목**

1. 오탈자 수정
   - 예: `병경` → `변경`
   - 예: 부자연스러운 Copilot 관련 표현 정리
2. 띄어쓰기·조사 보정
3. 위험 명령에 경고 표기 형식 통일
4. 명령어 예시 전후 설명 문장 톤 통일

## 5. 권장 작업 순서

```text
1단계: 02_git.md에서 Git 내부 모델 설명 추가 및 필요 시 보조 이미지 반영
2단계: 강의 전체에서 `git checkout` 사용 위치를 점검하고 `git switch`/`git restore` 기준으로 정리
3단계: 06_reset_revert.md와 03_start_new_feature.md의 명령어/표현 일관성 정리
4단계: 04_conflict.md와 05_rebase_squash.md에 협업 안전성과 검증 단계 보강
5단계: 98_tip.md, 99_etc.md에 실무 안전 수칙과 맥락 설명 추가
6단계: 전체 문서 오탈자/표현 정리
```

이 순서를 권장하는 이유는, 먼저 핵심 개념 축을 세운 뒤 `git checkout` 계열 명령을 전 문서 기준으로 먼저 정리해야 이후 개별 문서 수정이 충돌하지 않기 때문이다. 그 다음 각 문서의 안전성 설명을 보강하고, 마지막에 부록과 세부 품질을 정리하는 편이 전체 일관성을 유지하기 쉽다.

## 6. 검토 기준

수정 작업 후에는 아래 기준으로 점검한다.

### 내용 기준

- 같은 개념에 대해 서로 다른 명령어가 충돌 없이 설명되는가
- `git checkout`, `git switch`, `git restore`의 역할 구분이 분명한가
- 로컬 작업과 원격 협업 상황이 명확히 구분되는가
- 위험한 명령에 충분한 주의 문구가 있는가
- 초보자가 읽었을 때 “왜 이 명령을 쓰는지” 이해 가능한가

### 문서 기준

- 문서 간 용어와 표현 톤이 일관적인가
- 실습 흐름이 끊기지 않는가
- 부록이 본편과 자연스럽게 연결되는가
- 오탈자와 깨진 표현이 없는가

## 7. 이번 계획에서 제외하는 항목

이번 후속 계획에서는 아래 항목은 우선 제외한다.

- 강의 전체 순서의 대규모 재배치
- 새 장을 대량 추가하는 커리큘럼 확장
- 이미지 자산 전면 재제작

지금 단계에서는 범위를 넓히기보다, 기존 강의의 완성도를 높이는 쪽이 효과적이다.

## 8. 최종 정리

이 계획의 핵심은 현재 강의의 장점인 **실습성, 실무성, 흐름의 자연스러움**은 유지하면서, 초보자가 혼동하거나 실수할 수 있는 지점을 먼저 제거하는 것이다.

따라서 후속 수정은 “내용 추가”보다도 **일관성 정리, 안전성 보강, 설명 연결 강화**에 초점을 맞춰 진행하는 것이 가장 적절하다.
