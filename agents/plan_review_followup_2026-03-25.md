# Git 강의 자료 재리뷰 후속 계획서

> 작성일: 2026-03-25  
> 근거 문서: `agents/review_2026-03-25.md`

---

## 1. 목적

이번 계획의 목적은 재리뷰에서 남은 개선 후보로 정리된 항목들을 실제 수정 작업으로 옮길 수 있도록 구체화하는 것이다.

현재 강의의 큰 구조나 순서는 이미 안정적이므로, 이번 후속 작업은 전면 개편이 아니라 **마감 품질을 높이는 보완 작업**에 가깝다.

핵심 목표는 다음과 같다.

- 본편과 부록 사이의 맥락 연결 강화
- 남아 있는 구식 명령 표현의 현대화
- 환경/인증 안내의 최신성 보강
- 문서 간 이동이 더 자연스럽도록 교차 참조 추가

---

## 2. 범위

이번 계획은 아래 항목만 다룬다.

1. `01_quick_setup.md` 인증 파트 보강
2. `98_tip.md`의 stash 명령 표현 현대화
3. 본편 문서 간 짧은 연결 문장 추가
4. 부록 문서의 위치/성격 안내 보강
5. 소규모 설명 보강 (`04_conflict.md`, `05_rebase_squash.md`)

이번 계획에서는 아래 항목은 제외한다.

- 강의 순서 재배치
- 장 추가 또는 파일 대규모 분리
- 기존 본문을 크게 다시 쓰는 수준의 개편

## 4. 파일별 수정 계획

### 4-1. `01_quick_setup.md`

**목표**  
실습 환경 준비 문서의 인증 안내를 현재 사용 흐름에 더 가깝게 보완한다.

**수정 항목**

1. `Personal access tokens (classic)` 설명을 유지하되, 이것이 한 가지 방법임을 명시
2. SSH 키 방식도 많이 사용된다는 짧은 보조 설명 추가
3. 본편 진입을 돕는 연결 문장 추가
   - 예: “실습 환경 준비가 끝났다면 다음 문서에서 Git과 GitHub의 차이부터 살펴봅니다.”

**기대 효과**

- 환경 준비 문서가 현재 개발 환경 감각과 더 잘 맞게 됨
- `02_git.md`로 넘어가는 흐름이 부드러워짐

---

### 4-2. `03_start_new_feature.md`

**목표**  
본편의 출발 문서에서 다음 문서로 넘어가는 연결감을 더 높인다.

**수정 항목**

1. 문서 말미 또는 merge/conflict 관련 부분에 `04_conflict.md`를 자연스럽게 가리키는 문장 추가
   - 예: “실제로 충돌이 나는 경우는 다음 문서에서 직접 실습합니다.”
2. 필요 시 stash를 `98_tip.md`와 연결하는 짧은 참고 문장 추가
   - 예: “작업 도중 잠깐 다른 브랜치로 이동해야 할 때는 `98_tip.md`의 stash도 참고”

**기대 효과**

- 기본 작업 흐름에서 협업 문제 해결 문서로 자연스럽게 이어짐

---

### 4-3. `04_conflict.md`

**목표**  
충돌 문서의 초반 설명을 더 친절하게 만들고, 다음 문서와의 연결을 추가한다.

**수정 항목**

1. 맨 위 예시 코드 블록(`git merge main`)이 어느 브랜치에서 실행되는 상황인지 한 줄 설명 추가
2. 문서 말미에 `05_rebase_squash.md`와 연결되는 짧은 안내 문장 추가
   - 예: “충돌을 줄이고 커밋을 정리하는 방법은 다음 문서의 rebase/squash에서 이어집니다.”

**기대 효과**

- 초보자가 예시 상황을 더 쉽게 이해함
- conflict → rebase/squash 흐름이 더 자연스러워짐

---

### 4-4. `05_rebase_squash.md`

**목표**  
rebase 예시의 실행 맥락을 더 분명히 하고, 다음 문서로 이어지는 연결을 추가한다.

**수정 항목**

1. `git rebase main` 예시 앞뒤에 “현재 브랜치는 내 작업 브랜치여야 함”을 명시
2. 문서 말미에 `06_reset_revert.md`로 이어지는 연결 문장 추가
   - 예: “잘못된 변경을 되돌리는 방법은 `06_reset_revert.md`에서 이어집니다.”

**기대 효과**

- rebase 실행 위치에 대한 초보자 혼란을 줄임
- 이력 정리 → 복구 전략으로 흐름이 자연스럽게 이어짐

---

### 4-5. `98_tip.md`

**목표**  
부록 문서의 명령어 표현을 현재 강의 톤에 맞추고, 부록 문서 성격을 더 분명히 한다.

**수정 항목**

1. `git stash save "작업 내용 설명"`을 최신 표현 중심으로 조정
   - 권장 예시: `git stash push -m "작업 내용 설명"`
2. `git stash` / `git stash push -m` / `git stash pop` / `git stash apply` 흐름을 더 자연스럽게 재배치
3. `git push --force-with-lease`가 왜 `--force`보다 안전한지 한 문장 추가
4. 문서 서두에 “본편 학습 후 읽으면 좋은 실무 보조 기능” 같은 안내 문장 추가

**기대 효과**

- 부록 문서도 최신 Git 학습 흐름과 톤이 맞춰짐
- 수강생이 `98_tip.md`를 본편 이후 확장 학습 자료로 받아들이기 쉬워짐

---

### 4-6. `99_etc.md`

**목표**  
부록 문서의 성격을 조금 더 분명히 드러낸다.

**수정 항목**

1. 문서 상단 또는 첫 섹션 앞에 “참고/부록 성격의 문서”라는 짧은 안내 문장 추가
2. 본편 학습 이후 읽으면 더 좋다는 문맥 보강

**기대 효과**

- 본편과 부록의 역할 구분이 더 선명해짐

---

## 5. 권장 작업 순서

```text
1단계: 98_tip.md의 stash/force-with-lease 설명 현대화
2단계: 03_start_new_feature.md, 04_conflict.md, 05_rebase_squash.md에 교차 참조 추가
3단계: 01_quick_setup.md의 인증 파트와 본편 연결 문장 보강
4단계: 98_tip.md, 99_etc.md의 부록 성격 안내 문장 추가
5단계: 전체 표현 톤과 연결 문장 어색함 최종 점검
```

이 순서를 권장하는 이유는, 먼저 명령어 최신화와 본편 흐름 연결을 정리한 뒤, 마지막에 환경 안내와 부록 성격을 다듬는 편이 전체 사용자 경험에 더 직접적인 영향을 주기 때문이다.

## 6. 검토 기준

### 내용 기준

- `git stash` 관련 설명이 현재 강의의 현대적 명령어 톤과 어울리는가
- 본편 문서가 다음 문서로 자연스럽게 이어지는가
- 인증 안내가 너무 구식으로 느껴지지 않는가
- `--force-with-lease`의 의미를 초보자가 이해할 수 있는가

### 문서 기준

- 새로 넣은 연결 문장이 과하지 않고 자연스러운가
- 부록 문서가 본편을 방해하지 않으면서도 역할이 분명한가
- 기존 문서의 실습 중심 스타일을 해치지 않는가

---

## 7. 최종 정리

이번 후속 계획은 큰 구조 변경이 아니라, 이미 잘 정리된 강의를 **더 매끄럽고 더 현대적이며 더 안내 친화적인 자료**로 마감하는 단계다.

즉, 우선순위는 “새 내용을 많이 넣는 것”이 아니라, **흐름 연결, 최신 표현 정리, 안내 문장 보강**에 있다.
